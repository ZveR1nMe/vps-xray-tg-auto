# VPS Setup Script — Design Specification

## Overview

Bash-скрипт для настройки личного Ubuntu VPS: 3X-UI + VLESS Reality, hardening, Telegram-бот для управления. Запуск: `bash <(curl -sL <url>/setup.sh)`.

**Архитектура:** Bash-установщик + Python Telegram-бот (aiogram 3) как systemd-сервис.
**Целевое использование:** Личный сервер + друзья. Без домена. Только Reality.
**Клиент для пользователей:** Hiddify (Android, iOS, Windows, macOS) — бот генерирует `vless://` ссылку, друг вставляет через «Буфер обмена». Маршрутизация РФ-трафика напрямую — встроенный пресет Hiddify (Регион: Россия).

## 1. Структура файлов на сервере

```
/opt/vps-setup/
├── bot/
│   ├── bot.py                # Точка входа Telegram-бота
│   ├── handlers/
│   │   ├── status.py         # Статус сервера (CPU, RAM, диск, аптайм)
│   │   ├── traffic.py        # Трафик по пользователям
│   │   ├── users.py          # Управление пользователями (add/del)
│   │   ├── network.py        # Тест пинга, потери пакетов
│   │   ├── diagnostics.py    # Автодиагностика блокировок
│   │   ├── backup.py         # Бэкап конфигов
│   │   └── tips.py           # Советы по настройке, обход блокировок
│   ├── services/
│   │   ├── xui_api.py        # Взаимодействие с 3X-UI API
│   │   ├── link_gen.py       # Генерация vless:// ссылок
│   │   └── monitor.py        # Фоновый мониторинг + алерты
│   ├── config.py             # Загрузка конфига из .env
│   └── requirements.txt      # aiogram, psutil, aiohttp, python-dotenv
├── .env                      # Секреты (токен, chat_id, пароль панели) — chmod 600
├── data/
│   └── backups/              # Ротируемые бэкапы
└── logs/                     # Логи бота
```

### Хранение секретов

- Telegram Bot Token и Chat ID вшиты в `setup.sh` как константы (личный скрипт, не публикуется)
- При установке записываются в `/opt/vps-setup/.env` вместе с паролем панели 3X-UI
- `.env` — права `600`, владелец `root`
- `config.py` читает через `python-dotenv`

### Systemd-сервис

- `vps-bot.service` — Telegram-бот (автостарт, `Restart=always`)
- Бот работает от root (личный сервер, упрощает доступ к systemctl, логам, бэкапам)

### Установщик

Telegram Bot Token и Chat ID задаются как константы в начале `setup.sh`:
```bash
# ===== НАСТРОЙКИ — заполнить перед первым запуском =====
BOT_TOKEN="ВСТАВЬ_ТОКЕН_СЮДА"
CHAT_ID="ВСТАВЬ_CHAT_ID_СЮДА"
```

Перед первым использованием:
1. Создать бота через @BotFather → получить токен
2. Узнать свой Chat ID (написать @userinfobot или @getmyid_bot)
3. Вставить оба значения в скрипт

При запуске скрипт проверяет что значения заполнены (не плейсхолдеры) и отправляет тестовое сообщение. Если ошибка — останавливается.

`setup.sh` спрашивает только:
1. SSH-порт (предлагает сменить со стандартного 22)

Всё остальное — автоматически.

## 2. Настройка системы и безопасность

### Обновление системы

- `apt update && apt upgrade -y`
- Зависимости: `curl, wget, jq, python3, python3-pip, python3-venv, ufw, fail2ban, unzip`
- Автообновления безопасности через `unattended-upgrades`

### Файрвол (UFW)

```
ОТКРЫТО:
  - SSH-порт (кастомный)
  - 443/tcp — xray Reality

ЗАКРЫТО (всё остальное):
  - 2053 — порт панели 3X-UI (доступ только через SSH-туннель)
  - Все остальные порты — drop по умолчанию
```

### SSH-hardening

- Смена порта (пользователь задаёт при установке)
- Отключение `PasswordAuthentication` (только если `~/.ssh/authorized_keys` существует и непустой, иначе предупреждение и пропуск)
- `PermitRootLogin prohibit-password`
- `MaxAuthTries 3`

### Fail2ban

- Jail для SSH (5 попыток → бан на 1 час)
- Порт 443 не защищается fail2ban — xray Reality сам обрабатывает невалидные подключения (отдаёт ответ от dest-сайта)

### Оптимизация сети (sysctl)

```
net.core.default_qdisc = fq
net.ipv4.tcp_congestion_control = bbr
net.ipv4.tcp_fastopen = 3
net.ipv4.tcp_max_syn_backlog = 8192
net.core.somaxconn = 8192
net.ipv4.ip_forward = 1
```

### Доступ к панели 3X-UI

Только через SSH-туннель:
```bash
ssh -L 2053:127.0.0.1:2053 -p <ssh-port> root@<server-ip>
# Затем открыть http://localhost:2053 в браузере
```

Бот умеет напоминать эту команду.

## 3. 3X-UI и VLESS Reality

### Установка 3X-UI

- Официальный скрипт установки
- Панель слушает только `127.0.0.1:2053`
- Генерация случайного пути панели (`/secretpath-<random>`)
- Смена дефолтных логина и пароля на сгенерированные (сохраняются в `.env`)

### VLESS + Reality

```
Порт: 443
Flow: xtls-rprx-vision
Security: reality
SNI: выбирается из [www.microsoft.com, www.google.com, www.yahoo.com,
     www.apple.com, www.amazon.com] — скрипт делает outbound TCP connect test
     (curl --connect-timeout 3 -s -o /dev/null -w "%{time_connect}" https://<sni>)
     3 раза к каждому, выбирает с минимальным средним. Выполняется ДО запуска xray.
     Fallback: www.microsoft.com
Fingerprint: chrome
```

- Единственный протокол, без WS+TLS
- Не нужен домен и сертификат
- Максимальная производительность
- Reality маскирует трафик под обращение к легитимному сайту

### Конфиг xray

Один inbound на порту 443:
```json
{
  "inbounds": [{
    "tag": "vless-reality",
    "port": 443,
    "protocol": "vless",
    "settings": {
      "clients": [],
      "decryption": "none"
    },
    "streamSettings": {
      "network": "tcp",
      "security": "reality",
      "realitySettings": {
        "show": false,
        "dest": "<best-sni>:443",
        "xver": 0,
        "serverNames": ["<best-sni>"],
        "privateKey": "<generated>",
        "shortIds": ["<generated>"]
      }
    }
  }]
}
```

## 4. Telegram-бот

### Технологии

- Python 3 + aiogram 3
- psutil для системных метрик
- aiohttp для HTTP-запросов к 3X-UI API и check-host.net
- systemd-сервис с `Restart=always`

### Безопасность

- Привязан к одному `chat_id`
- Все команды от других пользователей — игнор

### 3X-UI API

Базовый URL: `http://127.0.0.1:2053/<panel-path>/`

**Аутентификация:**
- `POST /login` — body: `{"username": "...", "password": "..."}` → Set-Cookie с session
- При 401 — автоматический re-login
- При смене пароля — бот уведомляет о потере доступа

**Используемые endpoints:**
- `POST /xui/API/inbounds/list` — список inbound'ов с клиентами и трафиком
- `POST /xui/API/inbounds/addClient` — добавление клиента:
  ```json
  {
    "id": <inbound-id>,
    "settings": "{\"clients\":[{\"id\":\"<uuid>\",\"flow\":\"xtls-rprx-vision\",\"email\":\"<name>\",\"totalGB\":0,\"expiryTime\":0}]}"
  }
  ```
- `POST /xui/API/inbounds/<inbound-id>/delClient/<client-uuid>` — удаление клиента (endpoint может отличаться по версиям: проверить `clientId` vs `email` при реализации)
- `POST /xui/API/inbounds/getClientTraffics/<email>` — трафик по клиенту
- `GET /server/status` — статус сервера (CPU, mem, uptime)

Примечание: API 3X-UI не имеет официальной документации и может меняться. `xui_api.py` инкапсулирует все вызовы — при смене API нужно обновить только этот файл.

### Функции (inline-кнопки)

**Статус сервера:**
- CPU, RAM, диск, swap, аптайм, load average
- Версии ядра, 3X-UI, xray

**Сеть:**
- Пинг до Google DNS, Cloudflare, Moscow IX
- Потеря пакетов (серия ping 20 пакетов)
- Текущая скорость in/out

**Пользователи:**
- Список с трафиком (из 3X-UI API)
- Добавить друга → создаёт клиента в 3X-UI → генерирует `vless://` ссылку → отправляет в чат владельцу. Владелец пересылает ссылку другу вручную через Telegram.
- Удалить пользователя

**Диагностика блокировок:**
- Проверка доступности сервера из РФ (check-host.net API, fallback: isitdown.site)
- При недоступности обоих API — сообщение что диагностика временно недоступна
- Если IP заблокирован — совет сменить IP у провайдера
- Проверка что xray-сервис работает

**Советы:**
- Инструкция по установке Hiddify (ссылки для Android, iOS, Windows, macOS)
- Как вставить ссылку через «Буфер обмена»
- Как проверить что регион «Россия» выбран в маршрутизации
- Инструкция по SSH-туннелю к панели
- Какой SNI лучше для Reality

**Бэкапы:**
- По команде — архив в Telegram
- Ежедневный автобэкап в чат
- В архив: БД 3X-UI (`x-ui.db`), xray-конфиг. Логи и `.env` **не** включаются
- При пересоздании сервера — запустить `setup.sh` заново (токен и chat_id вшиты в скрипт, пароль панели генерируется новый)
- При размере > 45 МБ — только на сервер, уведомление в чат
- Ротация: хранятся 7 последних бэкапов, при создании 8-го удаляется старейший

**Алерты (фоновый мониторинг, каждые 5 минут):**
- CPU > 90% более 5 минут
- Диск > 85%
- Сервис x-ui или xray упал
- Аномальный трафик (текущий 5-минутный объём > 3x от среднего 5-минутного объёма за последние 24 часа; метрика: байты на сетевом интерфейсе через psutil)

## 5. Подключение друзей

### Генерация ссылки

При добавлении пользователя бот:
1. Создаёт клиента в 3X-UI через API (UUID, email)
2. Генерирует `vless://` ссылку в формате:
   ```
   vless://<uuid>@<server-ip>:443?type=tcp&security=reality&fp=chrome&pbk=<public-key>&sid=<short-id>&sni=<sni>&flow=xtls-rprx-vision#<name>
   ```
3. Отправляет ссылку в Telegram-чат

### Что делает друг

1. Устанавливает Hiddify (бот присылает ссылки на скачивание по платформам)
2. Копирует `vless://` ссылку из Telegram
3. В Hiddify нажимает «+» → «Буфер обмена» → готово
4. Маршрутизация РФ-трафика напрямую работает из коробки — Hiddify имеет встроенный пресет «Регион: Россия (ru)» в настройках маршрутизации

### Geo-данные

- На сервере — xray через 3X-UI использует встроенные `geoip.dat`/`geosite.dat`
- На клиенте — Hiddify использует свои встроенные geo-базы и обновляет автоматически
- Отдельное скачивание или обслуживание geo-файлов **не требуется**

## 6. Автообновления и cron

### Расписание

```
Ежедневно 04:00 UTC — бэкап конфигов → отправка в Telegram
Автоматически   — unattended-upgrades (только security)
Каждые 5 минут  — мониторинг (внутри бота, не cron)
```

### Автообновление 3X-UI

- Не автоматическое
- Бот раз в сутки проверяет новую версию через GitHub API
- Уведомление с кнопками: «Обновить» / «Пропустить»
- При обновлении: бэкап БД и бинарника → обновление → 10 сек ожидание → проверка `systemctl is-active x-ui` + HTTP-запрос к API → при ошибке → откат (восстановление БД и бинарника из бэкапа, перезапуск) → уведомление
- Откат всегда выполняется вслепую (без проверки миграции схемы). Если после отката сервис не запустился — бот сообщает и предлагает ручное вмешательство. Это крайне редкий сценарий.

### Логирование

- Cron-задачи пишут в `/var/log/vps-setup/`
- Ротация через logrotate (7 дней)

## 7. Деинсталляция

Команда: `bash /opt/vps-setup/setup.sh --uninstall`

- Останавливает и удаляет `vps-bot.service`
- Удаляет `/opt/vps-setup/` (с подтверждением)
- Удаляет cron-задачи
- **Не трогает:** 3X-UI, UFW-правила, SSH-конфиг, sysctl
- Выводит список оставшихся компонентов для ручной очистки

## Решения и trade-offs

| Решение | Почему |
|---------|--------|
| Bash + Python | Bash для сисадминки, Python для логики бота |
| Только Reality | Максимальная производительность, не нужен домен |
| vless:// ссылка через Telegram | Друг копирует → вставляет в Hiddify → всё работает |
| Hiddify как клиент | Кроссплатформенный, встроенный пресет маршрутизации РФ, бесплатный |
| Панель через SSH-туннель | Порт закрыт снаружи, максимальная безопасность |
| Клиентская маршрутизация | RU-трафик не делает крюк через сервер |
| Бот от root | Личный сервер, упрощает архитектуру |
| aiogram 3 | Async, inline-кнопки, стандарт |
